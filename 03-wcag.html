WCAG

Web Content Accessibility Guidelines

Niveaux de conformité

  • Niveau A : Critères de base pour l'accessibilité.
  • Niveau AA : Critères intermédiaires, visant à résoudre les problèmes d'accessibilité les plus courants.
  • Niveau AAA : Critères avancés, visant à offrir une accessibilité optimale.
Niveaux de conformité WCAG

Principes et critères

  • Perceptible : Le contenu doit être présenté de manière à ce qu'il puisse être perçu par tous.
  • Utilisable : Les fonctionnalités du site doivent être utilisables par tous, y compris ceux qui utilisent des technologies d'assistance.
  • Compréhensible : Le contenu et l'interface utilisateur doivent être compréhensibles pour tous.
  • Robuste : Le contenu doit être suffisamment robuste pour fonctionner avec diverses technologies, y compris les technologies d'assistance.
WCAG Quick Reference

1.4.1 : Utilisation de la couleur (A)

Le contenu visuel ne doit pas reposer uniquement sur la couleur pour transmettre des informations, indiquer une action, demander une réponse ou distinguer un élément visuel.

Problème :

/* Lien distingué uniquement
   par la couleur */
a {
    color: blue;
    text-decoration: none;
}

Solution :

/* Ajouter un soulignement */
a {
    color: blue;
    text-decoration: underline;
}

Les personnes daltoniennes ne peuvent pas distinguer un lien du texte normal si seule la couleur change.

WCAG 1.4.1 - Use of Color

1.3.1 : Information et relations (A)

L'information, la structure et les relations véhiculées par la présentation doivent pouvoir être déterminées par programmation.

Problème :

<!-- Tableau sans en-têtes -->
<table>
  <tr>
    <td>Nom</td>
    <td>Prix</td>
  </tr>
  <tr>
    <td>Produit A</td>
    <td>10 €</td>
  </tr>
</table>

Solution :

<!-- Utiliser th pour les en-têtes -->
<table>
  <thead>
    <tr>
      <th scope="col">Nom</th>
      <th scope="col">Prix</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>Produit A</td>
      <td>10 €</td>
    </tr>
  </tbody>
</table>
WCAG 1.3.1 - Info and Relationships

2.1.1 : Clavier (A)

Toutes les fonctionnalités du contenu doivent être utilisables au clavier sans exiger de contrainte temporelle particulière pour les frappes individuelles.

Problème :

/* Menu accessible uniquement au survol */
.menu:hover .submenu {
    display: block;
}

Solution :

/* Ajouter le support du focus */
.menu:hover .submenu,
.menu:focus-within .submenu {
    display: block;
}
WCAG 2.1.1 - Keyboard

2.4.4 : Fonction du lien (A)

La fonction de chaque lien doit pouvoir être déterminée par le texte du lien seul, ou par le texte du lien associé à son contexte.

Problème :

<!-- Liens non explicites -->
<a href="/article-1">Cliquez ici</a>
<a href="/article-2">En savoir plus</a>
<a href="/article-3">Lire la suite</a>

Solution :

<!-- Liens explicites -->
<a href="/article-1">
  Lire l'article sur l'accessibilité
</a>

<!-- Ou avec aria-label -->
<a href="/article-2"
   aria-label="En savoir plus sur WCAG">
  En savoir plus
</a>
WCAG 2.4.4 - Link Purpose (In Context)

4.1.2 : Nom, rôle et valeur (A)

Pour tout composant d'interface, le nom et le rôle doivent pouvoir être déterminés par programmation. Préférer les éléments HTML natifs.

Problème :

<!-- Div cliquable sans sémantique -->
<div class="btn" onclick="submit()">
  Envoyer
</div>

Solution :

<!-- Utiliser un vrai bouton -->
<button type="submit">
  Envoyer
</button>

<!-- Ou ajouter les attributs ARIA
     handleKey gère les touches Entrée/Espace -->
<div role="button" tabindex="0"
     onclick="submit()"
     onkeydown="handleKey(event)">
  Envoyer
</div>
WCAG 4.1.2 - Name, Role, Value

3.2.2 : À la saisie (A)

Le changement de paramètre d'un composant d'interface ne doit pas initier automatiquement un changement de contexte, sauf si l'utilisateur a été informé avant.

Problème :

<!-- Soumission automatique -->
<select onchange="this.form.submit()">
  <option>France</option>
  <option>Belgique</option>
</select>

Solution :

<!-- Bouton de validation explicite -->
<select id="country">
  <option>France</option>
  <option>Belgique</option>
</select>
<button type="submit">Valider</button>
WCAG 3.2.2 - On Input

1.4.3 : Contraste minimum (AA)

Le texte et les images de texte doivent avoir un ratio de contraste d'au moins 4.5:1 (ou 3:1 pour le texte agrandi ≥ 18pt ou 14pt gras).

Problème :

/* Ratio 2.8:1 - insuffisant */
p {
    color: #999999;
    background: #ffffff;
}

Solution :

/* Ratio 7:1 - conforme AAA */
p {
    color: #595959;
    background: #ffffff;
}
WCAG 1.4.3 - Contrast (Minimum) WebAIM Contrast Checker

1.4.4 : Redimensionnement du texte (AA)

Le texte doit pouvoir être redimensionné jusqu'à 200% sans perte de contenu ou de fonctionnalité, sans recourir à une technologie d'assistance.

Problème :

/* Unités fixes */
body {
    font-size: 14px;
}
.container {
    width: 960px;
    height: 400px;
    overflow: hidden;
}

Solution :

/* Unités relatives */
body {
    font-size: 1rem;
}
.container {
    max-width: 60rem;
    min-height: 25rem;
    overflow: visible;
}
WCAG 1.4.4 - Resize Text

1.4.11 : Contraste des éléments non textuels (AA)

Les composants d'interface et les éléments graphiques doivent avoir un ratio de contraste d'au moins 3:1 avec les couleurs adjacentes.

Problème :

/* Bordure peu visible (ratio 1.5:1) */
input {
    border: 1px solid #cccccc;
    background: #ffffff;
}

Solution :

/* Bordure visible (ratio 3.5:1) */
input {
    border: 1px solid #767676;
    background: #ffffff;
}
WCAG 1.4.11 - Non-text Contrast

2.4.6 : En-têtes et étiquettes (AA)

Les en-têtes et les étiquettes doivent décrire le sujet ou le but. La hiérarchie des titres doit être respectée (ne pas sauter de niveaux).

Problème :

<!-- Niveaux sautés -->
<h1>Accueil</h1>
<h3>Nos services</h3>
<h5>Service A</h5>

Solution :

<!-- Hiérarchie respectée -->
<h1>Accueil</h1>
<h2>Nos services</h2>
<h3>Service A</h3>
WCAG 2.4.6 - Headings and Labels

2.4.7 : Visibilité du focus (AA)

Toute interface utilisable au clavier doit avoir un mode de fonctionnement où l'indicateur de focus clavier est visible.

Problème :

/* Focus supprimé sans alternative */
button:focus {
    outline: none;
}

Solution :

/* Focus personnalisé visible */
button:focus-visible {
    outline: 3px solid #005fcc;
    outline-offset: 2px;
}
WCAG 2.4.7 - Focus Visible

Exemple : Target

En 2006, Target, une grande chaîne de magasins aux États-Unis, a été poursuivie en justice par la National Federation of the Blind (NFB) pour discrimination, car son site web n'était pas accessible aux personnes aveugles.

Les textes alternatifs (alt) des images n'étaient pas correctement renseignés, la navigation au clavier était difficile, et le design de la page de checkout ne permettait pas de distinguer le pointer de la souris.

En 2008, Target a accepté de régler le litige en payant 6 millions de dollars et en s'engageant à rendre son site web accessible en travaillant avec la NFB pour améliorer l'accessibilité de son site.

Example : Domino's pizza

En 2016, Guillermo Robles, une personne aveugle, a intenté une action en justice contre Domino's Pizza pour discrimination, car le site web de l'entreprise n'était pas accessible aux personnes utilisant des lecteurs d'écran.

Le site internet de Domino's Pizza ne respectait pas les normes d'accessibilité du WCAG :

  • Plusieurs images n'avaient pas de texte alternatif (alt).
  • Les liens n'étaient pas clairement identifiés (pas de texte associé).

Après plusieurs années de procédures judiciaires, la Cour suprême des États-Unis a statué en 2019 que les sites web des entreprises doivent être accessibles aux personnes handicapées, conformément à la loi sur les Américains handicapés (ADA).
