Outils

W3C

Le W3C propose une liste d'outils pour évaluer l'accessibilité des sites web. Ces outils peuvent être utilisés pour vérifier la conformité aux normes d'accessibilité.

Liste des outils d'évaluation de l'accessibilité du W3C

Il propose également un outil permettant de réaliser un rapport d'accessibilité en suivant les critères du WCAG :

Outil de rapport d'accessibilité du W3C

Cependant, cet outil ne réalise pas de vérification automatique, il est donc nécessaire de le compléter manuellement.

Dev Tools

La plupart des navigateurs modernes intègrent des outils de développement (DevTools) qui incluent des fonctionnalités pour évaluer l'accessibilité des pages web. Ces outils peuvent identifier certains problèmes d'accessibilité et fournir des recommandations pour les résoudre.

Dev Tools - Audit

L'onglet Audit (ou Lighthouse dans Chrome) permet de réaliser un audit d'accessibilité d'une page web. Il analyse divers aspects de la page, tels que la structure du contenu, les alternatives textuelles, la navigation au clavier, et fournit un rapport détaillé avec des suggestions d'amélioration.

Firefox intègre également un outil similaire appelé "Accessibility Inspector".

Dev Tools - Accessibility Tree

L'arbre d'accessibilité (Accessibility Tree) est une représentation hiérarchique des éléments d'une page web telle qu'elle est perçue par les technologies d'assistance. Il permet aux développeurs de comprendre comment leur contenu sera interprété par les lecteurs d'écran et autres outils d'assistance.

Il est basé sur le DOM (Document Object Model) de la page, mais simplifié pour ne montrer que les éléments pertinents pour l'accessibilité, avec des informations supplémentaires telles que les rôles ARIA, les états et les propriétés.

Extensions pour les navigateurs

  • WAVE (Web Accessibility Evaluation Tool) par WebAIM
  • axe DevTools par Deque Systems
  • Equal Access Toolkit par IBM
  • RGAA Checker

Limitations

Il est important de noter que les outils d'évaluation automatique de l'accessibilité ne peuvent pas garantir une conformité totale aux normes d'accessibilité. Ils peuvent identifier certains problèmes, mais une évaluation manuelle est souvent nécessaire pour une analyse complète.

Par exemple, différents outils permettent de détecter la présence d'alternatives textuelles pour les images, mais ils ne peuvent pas juger de la pertinence ou de la qualité de ces alternatives.

Limitations

Voici quelques exemples de critères du WCAG qui peuvent être complexes à évaluer automatiquement :

  • 1.1.1 Contenu non textuel : Vérifier que toutes les images ont des alternatives textuelles appropriées.

  • 1.3.1 Informations et relations : S'assurer que la structure du contenu est correctement marquée (titres, listes, tableaux).

  • 1.3.3 Caractéristiques sensorielles : Vérifier que les instructions ne dépendent pas uniquement de la forme, de la taille ou de la couleur.

  • 1.4.1 Utilisation de la couleur : S'assurer que les informations ne sont pas transmises uniquement par la couleur.

  • ...

De plus, l'accessibilité concerne également d'autres supports comme les vidéos, les documents PDF, les applications mobiles, etc., qui peuvent nécessiter des outils spécifiques pour une évaluation complète.

Testing automatisé en CI/CD

L'accessibilité peut être intégrée dans le workflow de développement via des outils de testing automatisé, exécutés à chaque commit ou pull request :

  • axe-core : Moteur de tests d'accessibilité utilisable en ligne de commande, dans Playwright, Cypress ou Selenium.
  • pa11y : Outil CLI qui analyse une URL et retourne les erreurs d'accessibilité. Facilement intégrable dans une pipeline CI/CD.
  • jest-axe : Matcher Jest basé sur axe-core, permettant de tester l'accessibilité des composants dans les tests unitaires.

Ces outils détectent automatiquement une partie des violations WCAG, mais ne remplacent pas un audit manuel.
